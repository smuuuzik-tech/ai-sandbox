## 1. Что такое Messages API и зачем он

Ты уже пользовался Claude через чат на claude.ai — это готовый продукт для конечного пользователя. API — это способ дотянуться до того же «мозга» Claude из своего кода.

Конкретно это одна функция:

```js
const response = await client.messages.create({
  model: 'claude-sonnet-4-5',
  max_tokens: 1024,
  messages: [
    { role: 'user', content: 'Привет' }
  ]
});
```

Всё что делают AI-продукты в мире — это вариации этого вызова с более хитрыми параметрами. Cursor, Perplexity, Khanmigo, Intercom Fin — все они внутри зовут такой же `messages.create({...})`. Отличаются только:

1. Системным промптом (роль и правила)
2. Что подкладывают в `messages` снаружи (код, веб-поиск, историю пользователя)
3. Параметрами `temperature`/`max_tokens`/`model` под задачу
4. Интерфейсом над этим

## 2. Роли: `user` и `assistant`

Массив `messages` состоит из объектов вида `{ role, content }`.

### Роль `user`

Это всё, что говоришь ты или твой код. Любой ввод — всегда под `'user'`.

### Роль `assistant`

Это реплики самой модели. **Сам API не кладёт ответ модели в `messages` автоматически.** Если хочешь продолжить разговор так, чтобы модель помнила предыдущий ход — тебе нужно самому положить её предыдущий ответ в `messages` под ролью `'assistant'`.

### Три железных правила

1. **Первое сообщение — `user`** (ты говоришь первым)
2. **Роли чередуются** (user → assistant → user → assistant → ...)
3. **Последнее сообщение — `user`** (кроме prefill-трюков, это Day 4)

### Что будет при нарушении

| Нарушение | Что получишь |
|---|---|
| Начал с `assistant` | `400 BadRequestError: first message must use the "user" role` |
| Два `user` подряд | `400 BadRequestError: roles must alternate` |
| Неизвестная роль (`bot`, `system` внутри messages) | `400 BadRequestError: role must be 'user' or 'assistant'` |
| Последнее — `assistant` | Модель не понимает на что отвечать (undefined behavior) |

### Типичная ошибка новичка

Хочешь передать модели несколько вопросов сразу и пишешь их отдельными сообщениями:

```js
// НЕПРАВИЛЬНО
messages: [
  { role: 'user', content: 'Привет' },
  { role: 'user', content: 'Какое сегодня число?' }
]
```

Правильно — склей их в одно сообщение:

```js
// ПРАВИЛЬНО
messages: [
  { role: 'user', content: 'Привет. Какое сегодня число?' }
]
```

## 3. `system` — самая мощная роль

**`system` — не сообщение в массиве `messages[]`. Это отдельный параметр верхнего уровня.**

```js
client.messages.create({
  model: 'claude-sonnet-4-5',
  max_tokens: 1024,
  system: 'Ты опытный редактор. Пишешь коротко и без воды.',   // ← отдельно
  messages: [
    { role: 'user', content: 'Отредактируй этот текст...' }
  ]
});
```

### Что кладут в `system` в реальных продуктах

- **Роль** — «Ты — психотерапевт», «Ты — финансовый аналитик»
- **Стиль** — «Отвечай на ты», «Без эмодзи», «Без вступлений»
- **Формат ответа** — «JSON со структурой `{answer, confidence}`», «3-5 пунктов»
- **Правила / ограничения** — «Никогда не советуй препараты. Направляй к врачу.»
- **Контекст продукта** — «Ты помощник в приложении для студентов 11 класса»
- **Примеры хороших ответов** (few-shot) — «Вот как надо: [пример 1]»

### Почему `system` сильнее, чем первый `user`-сообщение

Anthropic специально тренирует модель уделять **особое внимание** `system`. Этот параметр имеет больший вес, чем обычные сообщения. Если в `system` написано «отвечай только по-русски», а user пишет «answer in English» — Claude скорее послушает `system`.

### Примеры из реальных продуктов

**Cursor (AI-редактор кода, $9B оценка):**
```
Ты помощник по парному программированию в IDE.
- Подстраивайся под стиль существующего кода
- Перед тем как показать код, в 1-2 предложениях объясни что меняешь
- Если не хватает контекста — спрашивай, не гадай
```

**Perplexity (AI-поиск, $14B оценка):**
```
Ты Perplexity, research-ассистент.
- Для каждого фактического утверждения указывай источник
- Если не уверен в факте, напиши «У меня нет надёжной информации по этому»
- Никогда не спекулируй вместо того чтобы указать источник
```

**Khanmigo (AI-тьютор Khan Academy):**
```
Ты терпеливый учитель для школьника.
- Никогда не давай ответ напрямую. Веди ученика вопросами.
- Отмечай даже маленький прогресс.
- Подстраивай объяснение под предыдущие ответы ученика.
```

### Обязательная структура system prompt

Три части, которые покрывают 80% случаев:

1. **Кто ты** — роль
2. **Что ты делаешь** — задача
3. **Как ты это делаешь** — правила формата, тона, ограничений

### Технические детали

- `system` можно не указывать — модель ведёт себя дефолтно
- Длина `system` считается в input-токенах — платишь за неё при каждом запросе
- Один `system` на весь запрос, внутри `create()` не меняется

## 4. Multi-turn: разговор, который «помнит»

### Ключевой факт

**У Claude API нет памяти между запросами.** Ни байта. Каждый вызов `client.messages.create({...})` — изолирован от всех предыдущих.

Модель не знает:
- Что ты её спрашивал 5 минут назад
- Что она отвечала
- Кто ты и как тебя зовут

Каждый раз — чистый лист.

### Как тогда Cursor/ChatGPT «помнят»?

**Память полностью на стороне кода, не модели.** Ты сам сохраняешь всю переписку и подкладываешь её целиком в каждый следующий запрос.

**Ход 1:**
```js
messages: [
  { role: 'user', content: 'Объясни что такое async/await' }
]
```
Модель отвечает. Сохраняешь ответ в переменную `answer1`.

**Ход 2 — подкладываешь всё:**
```js
messages: [
  { role: 'user',      content: 'Объясни что такое async/await' },
  { role: 'assistant', content: answer1 },
  { role: 'user',      content: 'А в чём отличие от Promise?' }
]
```

Модель видит всю историю и может ссылаться на предыдущий контекст.

### Аналогия

Представь, что ты каждый раз нанимаешь нового консультанта с амнезией. Чтобы он «помнил» предыдущие разговоры — каждый раз приносишь распечатку всей прошлой беседы. Массив `messages[]` — это такая распечатка.

### Следствия multi-turn

**1. Разговор растёт по токенам с каждым ходом.**

- Ход 1: 50 токенов input
- Ход 2: 50 + 150 + 60 = 260 токенов input
- Ход 10: может быть 3000+ токенов

Ты платишь за все input-токены каждого запроса.

**2. У модели есть лимит context window.**

У Claude Sonnet 4.5 — **200 000 токенов**. Это ~150 000 слов или ~500 страниц текста. Для обычного чата хватает надолго, но в продуктах типа Cursor (с большим кодом) упираешься.

**3. В продакшне — управление контекстом.**

- **Обрезка** — выкидывать самые старые сообщения после N ходов
- **Суммаризация** — сжимать старую часть разговора в короткий пересказ
- **Prompt caching** — техника Anthropic (Day 8), где первые N токенов кешируются

На этапе обучения — не заморачивайся. Разговоры на 3-5 ходов стоят копейки.

## 5. Параметры через реальные продукты

### `model` — какую модель звать

| Модель | Мозги | Скорость | Цена input / output за 1M токенов |
|---|---|---|---|
| **Haiku 4.5** | Базовые | Быстрая | $1 / $5 |
| **Sonnet 4.5** | Хорошие | Средняя | $3 / $15 |
| **Opus 4.6** | Максимальные | Медленная | $15 / $75 |

Разница между Haiku и Opus — **15×** по цене. Ошибка в выборе модели напрямую бьёт по unit-экономике.

**Как решают реальные продукты:**

- **Intercom Fin** (AI для support-чатов) — Haiku для классификации типа вопроса, Sonnet только для сложных ответов. Экономия в разы.
- **Cursor** — Sonnet для обычного чата, Opus только для «agent mode» (планирование сложного рефакторинга).
- **Perplexity** — Sonnet для обычного ответа, Opus для платной подписки Pro.

**Правило:** начинай с **Sonnet** как default. Когда продукт заработал — смотри где можно заменить на Haiku без потери качества. Самая быстрая cost-оптимизация.

### `temperature` — разброс ответов

Число 0 ... 1. Не «умнее / тупее», а **стабильнее / креативнее**.

- **0** — детерминированно. Один и тот же запрос → буква в букву одинаковый ответ.
- **1** — максимум разброса. Каждый запуск — другая формулировка.

**Когда какое значение:**

| Задача | temperature | Почему |
|---|---|---|
| Классификация | **0** | Стабильность критична |
| Извлечение JSON | **0** | Любой разброс ломает парсер |
| Factual Q&A (Perplexity) | **0** | Правда одна |
| Поддержка клиентов (Intercom Fin) | **0–0.2** | Одинаковые кейсы — одинаковые ответы |
| Помощь с кодом (Cursor) | **0–0.3** | Код структурный, креатив вреден |
| Объяснения для студента | **0.3–0.5** | Чуть живости |
| Генерация идей/слоганов (Jasper) | **0.7–1.0** | Нужен разброс |
| Сторителлинг | **0.9–1.0** | Максимум творчества |

**Дефолт:** **0.7**. Не уверен — оставляй. Нестабильно — снижай. Скучно — поднимай.

### `max_tokens` — лимит длины ответа

Потолок генерации. Если модель считает что ответ короче — закончит сама (`stop_reason: end_turn`). Если поставишь мало — обрывает на полуслове (`stop_reason: max_tokens`).

**Как ставят:**

- SMS / уведомления: 150-300
- Чаты с людьми (Intercom): 500-1000
- Writing tools (Jasper): 2000-4000
- Code editors (Cursor): 4000-8000
- Research / анализ: 8000-16000

Подсказка: 1 токен ≈ 0.75 слов. `max_tokens: 1000` ≈ 750 русских слов ≈ 1.5 страницы A4.

### `stop_sequences` — редко

Массив строк. Модель оборвёт генерацию если сгенерит одну из них (`stop_reason: stop_sequence`). В 90% задач не нужен.

Полезен для ролевых агентов (не даёт модели говорить за собеседника) и для структурированных форматов с явным маркером конца.

### `top_p` — не трогай

Альтернатива `temperature`. Anthropic рекомендует оставить по умолчанию. Если встретишь совет крутить `top_p` — игнорируй.

## 6. Сводка блока 1

Теория Messages API целиком:

1. **API** — способ вызвать Claude из кода (`client.messages.create({...})`)
2. **Роли** — `user`/`assistant` чередуются, первое и последнее — `user`
3. **`system`** — отдельный параметр, задаёт роль и правила, сильнее чем user-сообщение
4. **Multi-turn** — API stateless, историю собираешь сам и шлёшь в `messages[]` целиком
5. **Параметры** — `model` (цена/мозги), `temperature` (разброс), `max_tokens` (лимит), `stop_sequences` (редко), `top_p` (не трогай)

**На что это всё ложится в реальности:**

Cursor, Perplexity, Intercom Fin, Khanmigo, Jasper — все это обёртки вокруг одного и того же API. Отличаются системным промптом, параметрами, что подкладывают снаружи в `messages[]`. Это то, что ты будешь повторять в своём продукте.
