### Что узнал
- Messages API работает через `client.messages.create({ model, max_tokens, messages })`
- Ответ приходит в `response.content[0].text`, а метаданные в `response.usage` и `response.stop_reason`
- Haiku быстрее Sonnet (у меня: 1395ms vs 1772ms на простом промпте)
- `stop_reason` бывает трёх видов: `end_turn` (норма), `max_tokens` (обрезалось), `tool_use` (дальше)
- Токены считаются и за input, и за output — платишь за оба

### Параметры которые играл
- **max_tokens=10** — модель обрывается на середине фразы, `stop_reason: max_tokens` кричит об этом
- **max_tokens=100** — нормально для коротких ответов типа слогана
- **temperature=0** — детерминированный ответ, при повторных запусках буква-в-букву одинаковый
- **temperature=0.7** — баланс, ответы меняются, но остаются в рамках
- **temperature=1** — максимум креатива и разброса
- **model=haiku** — дешевле, быстрее, хорошо для простых задач
- **model=sonnet** — баланс для большинства случаев (у меня сейчас default)
- **model=opus** — для сложных задач где надо максимум мозгов

## Day 2 — 22.04.2026 (факт: сделан 21.04, тот же день что Day 1)

### Messages API — полная картина

**Три роли:**
- `user` — сообщение от человека / твоего кода
- `assistant` — реплика модели
- `system` — **отдельный параметр** верхнего уровня (не внутри `messages[]`). Задаёт роль/стиль/правила. Не виден пользователю. Один на диалог.

**Multi-turn: у API НЕТ памяти.** Каждый `create()` — отдельный запрос. Чтобы модель помнила предыдущий ход — сам передаёшь всю историю в `messages[]`:
```js
messages: [
  { role: 'user',      content: 'вопрос 1' },
  { role: 'assistant', content: 'ответ 1' },
  { role: 'user',      content: 'вопрос 2' },
]
```
Следствие: каждый новый ход растёт по токенам. В продакшне делают context management (обрезают / суммируют старое).

**content.** В простом случае — строка. В продвинутом — массив блоков: `text`, `image`, `tool_use`, `tool_result`. Сегодня только строки.

### Параметры — таблица

| Параметр | Что делает | Когда важно |
|---|---|---|
| `model` | какую модель звать | Haiku простое / Sonnet default / Opus сложное |
| `system` | инструкция на весь диалог | Стиль, роль, правила, формат |
| `max_tokens` | лимит генерации | Мало → обрыв. Много → платишь за лишнее |
| `temperature` | 0...1 разброс | 0 = детерминированно. 1 = креатив |
| `stop_sequences` | строки, на которых оборвать | Редко. Когда нужно не ходить дальше маркера |
| `top_p` | альтернатива temperature | Обычно не трогай |

### Ключевые insights этого блока
- `system` ≠ элемент `messages[]`. Это отдельный top-level параметр. Частая ошибка — запихивать его первым user-сообщением.
- API stateless. Вся память — на моей стороне (я храню и переотправляю messages).
- `top_p` и `temperature` — два разных способа управлять разбросом. Крутить оба одновременно = запутаться. Правило: трогай только temperature.

### Сравнение Haiku / Sonnet / Opus на живом 3-ходовом диалоге

Одинаковый сценарий (наставник по macOS, 3 хода), один и тот же system prompt:

| Модель | Время | Токены in/out | Стоимость | Качество |
|---|---|---|---|---|
| Haiku 4.5 | 7.4s | 1170 / 441 | $0.003 | **ОШИБСЯ по факту** — сказал что 4 ГБ для Chrome это много (неверно) |
| Sonnet 4.5 | 16.2s | 1289 / 588 | $0.013 | Правильно, развёрнуто, держит наставнический тон |
| Opus 4.6 | 13.1s | 1231 / 434 | $0.051 | Правильно, максимально минималистично, лучший «Khanmigo-стиль» |

**Главный урок:** Haiku ненадёжен там где нужно суждение «норма/ненорма» или нетривиальный факт. Для простой классификации — ок. Для диагностики/совета — Sonnet минимум.

**Экстраполяция цены на 1000 разговоров/день:**
- Haiku → ~$90/мес
- Sonnet → ~$400/мес
- Opus → ~$1500/мес

**Стиль ответов отличается даже при одинаковом system prompt:**
- Haiku: командный, списки-шаблоны
- Sonnet: развёрнутый наставник, ведёт через вопросы
- Opus: минималистичный, механизм + один вопрос, ближе всех к «настоящему Khanmigo»

## Day 2, блок 3 — параметры (`params-playground.js`)

Один и тот же промпт, меняем только параметры. Три эксперимента в одном файле.

### Эксперимент 1 — temperature

По 3 запуска на temp=0, temp=0.7, temp=1. Задача: слоган для кофейни.

| temp | запуск 1 | запуск 2 | запуск 3 |
|---|---|---|---|
| 0   | «Где город встречается с кофе» | «Ваш идеальный перерыв в ритме города» | «Где город встречается с кофе» |
| 0.7 | «Где город встречается с кофе» | «В самом сердце города — для вашего сердца кофе» | «Где город встречается с кофе» |
| 1   | «Где город встречается с кофе» | «Вкус, который пробуждает город» | «"В сердце города — кофе для твоего сердца"» |

**КЛЮЧЕВОЙ ИНСАЙТ:** `temperature=0` у Claude API **не даёт 100% детерминизм**. На 3 запусках получил 2 одинаковых и 1 другой. Причина — floating-point + batching на стороне Anthropic (запрос может попасть в разный батч → чуть другие округления → другой токен). У Claude API **нет параметра `seed`** (в отличие от OpenAI). Строгой воспроизводимости получить нельзя.

Практика:
- temp=0 → сильное смещение в один вариант, но не гарантия буква-в-букву
- temp=1 → все три разные, разброс явно больше
- Для production-парсера не полагайся на «одинаковость» temp=0. Если нужна стабильность JSON — валидируй на стороне кода + retry на ошибках.

### Эксперимент 2 — max_tokens

Промпт «объясни что такое API», temperature=0, меняем только max_tokens.

| max_tokens | out токенов | stop_reason | что получил |
|---|---|---|---|
| 10  | 10  | max_tokens | «# API простыми словами\n\n**API» (обрыв на слове) |
| 50  | 50  | max_tokens | «...Представ» (обрыв) |
| 300 | 153 | end_turn   | полный ответ с аналогией про официанта |

Два урока:
1. `max_tokens` — это **лимит, не цель**. При 300 модель использовала только 153 — закончила сама. Платишь за фактические out-токены, не за потолок. Ставь с запасом, без страха переплатить.
2. Sonnet на простом «объясни» **самовольно добавил markdown** (# заголовок, **жирный**). Дефолт поведения Sonnet — структурировать. Чтобы выключить — в `system`: «Отвечай без markdown-форматирования».

### Эксперимент 3 — stop_sequences

Промпт: «5 слоганов нумерованным списком».
- Без стопа → 5 пунктов, `stop_reason: end_turn`
- Со `stop_sequences: ['3.']` → 2 пункта, `stop_reason: stop_sequence`. Модель физически не сгенерировала «3.» — встретила маркер, оборвалась.

Когда полезно:
- Ролевой агент: `stop_sequences: ['USER:']` — не даёт модели писать за пользователя
- Code gen: `stop_sequences: ['```']` — обрывает после закрывающих бэктиков
- Диалог-симуляция: `stop_sequences: ['\nОтвет:']` — не даёт модели отвечать за собеседника

В 90% обычных задач не нужен. Когда нужен — ничем не заменишь.

### Свод по параметрам
- `model` — осознанно. Ошибка = 15× по цене.
- `temperature` — 0 для парсинга, 0.7 дефолт, 1 для креатива. От temp=0 не жди полного детерминизма.
- `max_tokens` — ставь с запасом. `stop_reason: max_tokens` = мало дал.
- `stop_sequences` — редкий, но уникальный инструмент.
- `top_p` — не трогаешь.

## Day 3 — 21.04.2026 (календарно в тот же день, что Day 1+2)

### Концепт: tool use = мозг Claude + руки моего кода

Главная фраза дня. Запомнить как аксиому. Все AI-агенты (Cursor, Perplexity, Replit Agent, Devin, Claude Code, Intercom Fin) построены на этом принципе.

### Жизненный цикл tool use — пять шагов

1. Я шлю запрос с `messages` + `tools` (описания в JSON Schema).
2. Claude читает описания, решает: отвечать напрямую или вызвать tool. Если tool — возвращает ответ со `stop_reason: 'tool_use'` и блоком `tool_use` внутри `content` (там `name`, `input`, `id`).
3. Claude **ничего не исполняет**. Исполняю я: парсю блок, вызываю свою функцию с `input`, получаю результат.
4. Второй запрос к API: история + assistant-сообщение с оригинальным content первого ответа + **user-сообщение с блоком `tool_result`** (тот же `id` + мой результат).
5. Claude видит результат и генерирует финальный текст для пользователя (`stop_reason: 'end_turn'`).

### Контринтуитивные моменты

- **`tool_result` идёт с ролью `user`**, не assistant. Смысл: результат внешнего инструмента для API — это новая входная информация от среды, а не «речь модели».
- Ассистентский ответ с `tool_use` нужно пропихнуть в `messages` **ЦЕЛИКОМ** (response.content). Не парсишь обратно — именно сырой массив блоков. Иначе Claude не свяжет `tool_use_id` с `tool_result`.
- В каждом запросе цикла `tools` отдаю заново. API не держит их между запросами.
- Моё поле `caller: { type: 'direct' }` в tool_use — это новая мета-инфа SDK, связана с Claude skills. Для базового tool use игнорирую.

### Description — это продажа tool модели

От качества `description` зависит **когда** Claude позовёт инструмент. Плохое описание = модель зовёт tool не в те моменты или вообще пропускает. Писать description как объяснение джуну на ревью: «что делает, когда использовать, какие аргументы обязательны».

### Агентный цикл — прелюдия к Day 10

Когда tools зависят друг от друга (IP → локация → погода), один проход не решает задачу. Нужен `while (stop_reason === 'tool_use')` с safety-счётчиком итераций. Паттерн: **итераций = количество tool-вызовов + 1** (последняя — когда модель уже получила всё и пишет финальный текст).

Сегодня сделал на 3 tools и 4 итерации. На Day 10 пройдём это формально и глубже.

### Экономика цикла — главный урон по кошельку

На моём `tool-use-multi.js`:
- Итерация 1: 2481 in / 92 out
- Итерация 2: 2596 in / 63 out
- Итерация 3: 2723 in / 78 out
- Итерация 4: 2871 in / 217 out

**+100-150 токенов на каждой итерации** из-за накопления истории (assistant+tool_use + user+tool_result). Плюс `tools` шлются заново каждый раз.

Итого: $0.038 за ОДИН запрос агента. В продукте на 1000 DAU × 5 запросов в день = **$6000/мес только на агентный слой**. Prompt caching (Day 8) — не roskosh, а то без чего агенты экономически не живут.

### Sonnet по дефолту накидывает markdown + эмодзи

Повторное подтверждение инсайта Day 2. В финальном ответе `tool-use-multi.js` появились 🌡️ 🌧️ 💨 + жирный текст. Если продукт на SMS/Telegram-боте/voice — выключать в `system`: «Отвечай чистым текстом, без markdown и эмодзи».

### Free IP-геолокаторы не дружат с VPN-диапазонами

Сегодня **три раза переключал провайдера**:
- ipapi.co → `{"reason": "RateLimited", "error": true}`
- ipwho.is → `{"error": "CORS is not supported on the Free plan"}` (хотя CORS к Node.js не имеет отношения — это заглушка)
- freeipapi.com → **заработало**, отдал Хельсинки + `is_proxy: false`

Два практических вывода:
1. Бесплатные IP-API лимитируют/резают дата-центровые и прокси-диапазоны — это основной abuse-канал.
2. Поле `is_proxy` **не согласовано** между провайдерами. ipapi/ipwho фактически обозначили мой IP как прокси, а freeipapi написал `false`. Если в продукте будет бизнес-логика на `is_proxy` — нужно агрегировать несколько источников или брать платный anti-abuse-сервис (IPQualityScore).

### Главный поведенческий инсайт Sonnet сегодня

На первом прогоне `tool-use-multi.js` ipapi вернул пустую локацию. **Claude сам решил НЕ звать `get_weather_forecast` с пустыми/нулевыми координатами** — вместо этого вернул управление пользователю и спросил город. Это не прописано в коде, это встроенный reasoning модели. Важный фактор надёжности агентов: если `tool_result` возвращает ошибку явным полем `error: "..."`, Claude умеет это видеть и корректно обходиться.

### Где ценность в связке LLM + tools

API open-meteo вернул сухие цифры: `temp_max: 13.2, precipitation: 0.2mm, wind: 29.9`. Claude в финале написал: «прохладно, особенно утром, одеться теплее, зонт на всякий случай». **Интерпретация числа в человеческий совет — это и есть продуктовая ценность**. Голый API погоды не посоветует зонт. Этот же паттерн в Perplexity (факты из поиска → связный ответ), Replit Agent (вывод bash → «значит баг в X»), Intercom Fin (данные из KB → ответ клиенту).

## Day 4 — 22.04.2026. Structured Outputs + Zod-валидация

### Концепт дня

Structured output = ответ Claude строго по JSON Schema, а не свободным текстом. Без этого нет LLM-бэкенда, только демка. Реализация через **tool use с форсированным `tool_choice`** — заставляем модель гарантированно вызвать конкретный tool, её `input` и есть наш JSON. API Anthropic валидирует `input` по `input_schema` на своей стороне — мы получаем либо валидный объект, либо ошибку запроса.

Сегодняшний пример: парсер сырых клиентских обращений в структурированные баг-тикеты для баг-трекера (Linear/Jira-стиль).

### tool_choice — три режима

```js
// Claude сам решает
tool_choice: { type: 'auto' }

// обязан вызвать хоть какой-нибудь tool
tool_choice: { type: 'any' }

// обязан вызвать именно этот — НАШ ВАРИАНТ ДЛЯ STRUCTURED OUTPUTS
tool_choice: { type: 'tool', name: 'submit_bug_report' }
```

Для structured outputs — только третий. Первый и второй разрешают модели «ответить текстом», и тогда структуры нет.

### Принципы хорошей схемы

Разобраны на примере bug_report-схемы:

- **Enum там где домен ограничен** (`severity`, `component`). Без enum получаешь «High», «high», «важный», «🔴». Enum физически ограничивает модель набором токенов.
- **Nullable через `type: ['string','null']` + явная инструкция в description** («null если не указано, не выдумывай»). Без этой инструкции LLM выдумывают — выдумать кажется им менее плохим чем оставить пустым.
- **description = часть промпта, не документация**. Модель уважает описания полей сильнее чем `minLength/maxLength`. Пиши как ТЗ джуну.
- **Плоская структура лучше вложенной**. Чем глубже — тем чаще модель ошибается на глубоких полях.
- **required только по minimum** — остальное nullable/optional, иначе ложные провалы.
- **Self-evaluation поле** (`has_enough_info: boolean`) — модель оценивает собственный вывод. Паттерн Cursor / Perplexity / Claude Code. Даёт нам управляемую «неуверенность».

### Экономика — самая болезненная часть

На моей схеме с подробными описаниями:
- Input: ~6500 токенов (из них ~6400 — СХЕМА, реальный текст клиента ~100)
- Output: ~500-600 токенов (сам тикет)
- Стоимость: **~$0.028 за один тикет** на Sonnet 4.5

Экстраполяция 1000 тикетов/день = $28/день = **$840/мес** только на парсинг.

**Prompt caching (Day 8) — не оптимизация, а необходимость.** Схема и system неизменны между вызовами. Закешируем → экономия 80-90% → $84/мес вместо $840. Это разница между «можем себе позволить» и «стартап убивает».

### Границы применимости — ключевой урок

**1. Форсированный tool_choice не даёт модели отказаться.**
Кейс 5 — запрос фичи, не баг. Claude всё равно натянул на bug_report: `severity: low, component: ui`. Модель видит несоответствие, но `tool_choice: {type: 'tool'}` обязывает её вызвать именно этот tool. Вывод: для реальных продуктов нужна **двухступенчатая архитектура** — сначала классификация (`ticket_type`) на cheap-модели, потом роутинг в нужную схему. Так работают Intercom Fin, Zendesk AI, Sentry-AI.

**2. Escape hatch в непредусмотренном поле.**
В одном из прогонов Claude на том же кейсе 5 использовал `missing_info` не по назначению — написал туда «Это feature request, а не баг-репорт». Модель сама нашла способ передать мета-информацию через доступное поле. Плюс — гибкость, минус — непредсказуемо (в другом прогоне не сработало). Лечится явным добавлением нужных полей в схему.

**3. Межзапусковая вариативность даже со structured outputs.**
Подтверждение инсайта Day 2 про temperature. На одинаковых входах:
- severity в кейсе 1: `critical` → `high` → `high` → `high` (4 прогона)
- has_enough_info в кейсе 5: `true` → `false` → `true` (3 прогона)
- missing_info в кейсе 2: 5 → 6 пунктов

Схема стабилизирует ФОРМУ, не СОДЕРЖИМОЕ. В продакшне: метрики «critical багов в неделю» плавают не из-за реальной динамики, а из-за рулетки LLM. Для критичных решений нужен human-in-the-loop или ансамбль.

### Zod — валидация на клиенте поверх API

Зачем дубль, если API уже валидирует? **API гарантирует синтаксис, ты гарантируешь семантику.** API проверяет типы/enum/required. API НЕ проверяет:

- **связки между полями** — «если has_enough_info=true, то missing_info должен быть пустым». Claude нарушает это регулярно, модель видит это как рекомендацию, не правило.
- **бизнес-логику** — «critical-баг не может быть в компоненте other».
- **логические противоречия** — «стаб-шаги + has_enough_info=true = невозможно».

Zod добавляет это через `.refine((data) => condition, {message, path})`. Кидаешь функцию-валидатор, она возвращает boolean. `path` указывает какое поле семантически связано с ошибкой — важно для дебага.

### safeParse vs parse

- `parse(obj)` — бросает `ZodError` на невалидном. Нельзя использовать в батч-обработке: один битый тикет уронит весь пайплайн.
- `safeParse(obj)` — возвращает `{success, data | error}`. Не прерывает выполнение. **Для LLM-пайплайнов — только safeParse.**

### Что реально поймало у меня в прогоне

5 синтетически сломанных объектов, Zod поймал все 5 с точными сообщениями:
- Правило 1: has_enough_info=true + missing_info=[...] — `[missing_info]`
- Правило 2: severity=critical + component=other — `[component]`
- Правило 3: stub-шаги + has_enough_info=true — `[has_enough_info]`
- Базовая: severity='blocker' — `[severity] Invalid option`
- Базовая: title='Баг' (3 символа) — `[title]` короткий

На реальных 5 кейсах все 5 прошли валидацию — это норма при грамотной схеме. Zod — safety net для 1-5% аномалий, не primary mechanism.

## Day 5 — 22.04.2026

### Streaming (SSE — Server-Sent Events)

**Суть.** Обычный `messages.create()` блокирует код пока Claude не сгенерит весь ответ (5-10 секунд тишины для юзера). `messages.stream()` открывает HTTP-соединение и **не закрывает** — сервер пушит ответ кусочками по мере генерации. Реальная скорость не меняется, но **воспринимаемая скорость × 5** потому что первое слово появляется через ~1.5 сек вместо ~8 сек.

**Под капотом.** Поток SSE-событий: `message_start` → `content_block_start` → серия `content_block_delta` (text_delta кусочками или input_json_delta для tool use) → `content_block_stop` → `message_delta` (финальные output_tokens) → `message_stop`. SDK обёртывает это в 3 интерфейса:
- `stream.on('text', cb)` — самый высокий уровень, колбэк на готовые куски текста
- `for await (chunk of stream)` — async iterator, сырые события (нужно для tool use)
- `await stream.finalMessage()` — ждёт конца и возвращает собранное сообщение как обычный response

Эти три **не взаимоисключающие**. Стандартный prod-паттерн: `.on('text')` для печати в UI + `finalMessage()` для логирования.

**Где это в реальных продуктах.** Claude.ai/ChatGPT — стримят текст. Cursor — стримит inline-дополнения и diff через tool_use (`input_json_delta` парсится инкрементально). Perplexity — двухфазный стрим: сначала источники, потом ответ. v0.dev — стримит tool_use с контентом файла, справа HMR обновляет превью. Replit Agent — стримит лог действий.

### Где измерять в стриме — TTFT vs total latency

- **TTFT (time to first token)** — засекаем в момент прихода **первого** `text_delta`. Это UX-метрика: «через сколько юзер увидел реакцию».
- **Total latency** — от старта запроса до `finalMessage()`. Это инфраструктурная метрика.
- **Output_tokens можно узнать ТОЛЬКО после закрытия стрима** — приходит в `message_delta` последним. Стоимость считаем только после финала, не в процессе.

На моих замерах: TTFT 815-1840ms, total 7-15 сек в зависимости от длины ответа. **Стрим съедает ~90% воспринимаемого ожидания.**

### Error handling — категории ошибок и retry

**Три категории (по HTTP-коду):**
- **Транзиентные** (ретраим): 408, 429 rate limit, 500/502/503, 504, 529 overloaded + сетевые таймауты (APIConnectionError). Это «подожди и попробуй снова».
- **Фатальные** (НЕ ретраим): 401 auth, 403 forbidden, 400 bad request, 404. Ошибка в коде/конфиге, ретрай не поможет.
- **Серая зона**: обрыв стрима посреди — обычно ретраим с осторожностью.

**Exponential backoff with jitter.** Формула: `baseDelay × 2^(attempt-1) × (1 ± 0.3)`. У меня получилось: попытка 2 = 1183ms, попытка 3 = 1813ms (база 1000ms). Jitter критичен — без него при массовом сбое 1000 клиентов ретраят синхронно и добивают сервер. С jitter — ретраи размазаны во времени.

**SDK умеет сам.** Anthropic SDK ретраит 429/500/529 автоматически с backoff + уважает Retry-After header. Параметры: `maxRetries` (default 2), `timeout` (default 60_000ms). Для 90% кода этого достаточно. Своя обёртка нужна когда хочешь видеть ретраи в логах, делать fallback (Sonnet → Haiku при 529), или circuit breaker.

**Продакшн-правило.** Никогда не показывать юзеру 500 ошибку. Всегда retry + fallback-ответ. Как это делают: Cursor показывает «reconnecting...», Perplexity тихо переключается на fallback-модель, Intercom Fin возвращает «передаю оператору» при полном фейле.

### Observability — JSONL-логгер

**Формат — JSONL (JSON Lines).** Одна запись = одна строка валидного JSON + `\n`. Почему этот формат:
- Аппендится без перечитывания файла (важно для 1000+ записей/день)
- Каждая строка парсится независимо — одна битая не ломает всё
- Читается `jq`, `grep`, импортируется в BigQuery/ClickHouse напрямую
- Это де-факто стандарт для LLM-логов (Langfuse/Helicone тоже так)

**Что логировать на каждый вызов.** timestamp, model (точная версия), input_tokens, output_tokens, latency_ms, ttft_ms (для стрима), cost_usd, stop_reason, request_id (для запроса в Anthropic support), success, error (если упало), tag (что за сценарий), user_id (в проде).

**stop_reason: 'max_tokens' — это алерт.** На стриме я поставил max_tokens=300, Claude уткнулся и ответ обрезался. В проде процент таких запросов — первая метрика на дешборде. Означает «юзер получил неполный ответ».

**Request_id всегда сохраняем.** По нему Anthropic support может в своих логах посмотреть что именно произошло. Это единственный способ разобраться в странных сбоях.

**Продукты для observability (на будущее).** Langfuse (self-hosted Sentry для LLM с трейсингом chain), Helicone (прокси перед Anthropic, ловит всё), Braintrust (evals + логи). OpenTelemetry стандарт для LLM теперь есть — можно логировать в любой APM. На моём этапе (1 dev, 0 юзеров) — свой JSONL достаточно. Перейду на Langfuse когда будет 50+ юзеров.

### Инсайт: промпт не гарантирует поведение

### Концепт дня

Первый работающий **агентный цикл**: Claude в роли seed-инвестора прогоняет идею фаундера через интервью в стиле Mom Test, задаёт до 5 уточняющих вопросов, в конце вызывает terminator tool `deliver_verdict` с структурированным вердиктом (go/pivot/kill + 6 скоров + 3 риска + 3 валидации + kill criteria). ### Главная развилка: «можно ли это сделать в чате Claude?»

Начинал с Post Generator для TG/IG. В середине дня спросил себя этот вопрос — ответ «да, легко». Остановился. Вывел правило: **AI-API-скрипт оправдан только если присутствует хотя бы один из пяти признаков**:
1. **Триггер** — автоматический запуск по событию (cron, webhook, новый файл)
2. **Батч** — много объектов в цикле
3. **Интеграция** — подтягивает данные из другого сервиса
4. **Agent loop** — многошаговое рассуждение с tools
5. **Приватность** — данные нельзя отправить в веб-чат

Post Generator не проходил ни один критерий. Idea Pressure Tester проходит agent loop. Переориентировался.

### Архитектура agent loop — три файла + while-цикл

- `prompts/investor-skeptic.js` — system prompt (3KB) как отдельный артефакт. Роль, методология (Mom Test / YC triad / 4D fit), правила вопросов, тон. Паттерн «промпты как артефакты» — это reusable система, промпт живёт своей жизнью отдельно от логики.
- `schemas/verdict.js` — Zod-схема для локальной валидации + JSON Schema для `VERDICT_TOOL.input_schema` (что Claude реально читает). Пишу JSON Schema руками, не генерю из Zod — description'ы слишком важны, чтобы их доверять автогенератору.
- `idea-tester.js` — CLI + agent loop.

Цикл:
```js
while (true) {
  const response = await client.messages.create({...tools, messages});
  messages.push({ role: 'assistant', content: response.content }); // ЦЕЛИКОМ
  if (response.stop_reason === 'end_turn') break;
  const toolResults = response.content
    .filter(c => c.type === 'tool_use')
    .map(tu => handleTool(tu));
  messages.push({ role: 'user', content: toolResults });
  if (deliver_verdict_called) break;
  if (++iterations > MAX_ITER) break; // safety
}
```

### Два ключевых safety-паттерна

**1. Zod self-heal.** Первая ошибка валидации → кидаем Claude обратно `tool_result` с `is_error: true` и списком `path + message`. Модель получает шанс переделать. **Только одна попытка** — без ограничения легко уйти в бесконечную петлю «Claude исправляет одно, ломает другое». На двух прогонах обе ошибки Zod были исправлены с первой попытки.

**2. Budget-aware agent.** `MAX_CLARIFICATIONS = 5`. Когда бюджет кончается — инжектим в следующий user-turn системную заметку «бюджет вопросов исчерпан, вызывай `deliver_verdict` сейчас». Паттерн: управление агентом через инжекты в conversation, не через жёсткие rule-based остановки. Agent продолжает думать самостоятельно, просто с обновлённым контекстом.

### Tool description — это продажа tool'а модели

На `deliver_verdict` description начал с правила: «Call this ONLY when you have enough information to give a SPECIFIC, non-generic verdict. If the answers are still vague — ask one more clarification instead.» Это работает: Claude на первых 2-3 вопросах выбирал `ask_clarification`, не пытался схлопнуться в вердикт раньше времени. **Description — это не документация, это инструкция, которую модель читает в том же поле что и системный промпт.** Пиши её как ТЗ джуну.

### Извлечённый инсайт про собственную нишу

Claude в роли инвестора спросил: «назови 3-5 реальных клиентов Proxy.Market с именами и конкретными часами/бюджетом на поддержку парсеров». ### Unit economics — главная цифра дня

**$0.06/страница × 100 000 страниц = $5 961 ТОЛЬКО на Claude Sonnet.** Без Jina Premium, без proxy, без Playwright. Сравнение: Firecrawl $0.016/стр, мы в 4 раза дороже на голом pipeline.

- **Haiku вместо Sonnet** — input $1/M vs $3/M → ~$0.02/стр. Но качество на сложных страницах надо мерить.

### Self-reported confidence — паттерн, который работает
